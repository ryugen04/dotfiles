# 調査レポートフォーマット（絶対遵守）

NO INVESTIGATION REPORT WITHOUT QIITA-STYLE STRUCTURE

## 原則

**調査レポートを作成する際は、必ず `investigation-report` スキルに従う。**

事実の列挙だけで終わらせず、Qiita 記事レベルの粒度で「初見の読者が地図を描けるレポート」を作ること。

## 対象

| ファイル種別 | 対象 |
|-------------|------|
| 調査レポート | `.claude/work/**/investigation*.md`, `.claude/work/**/article*.md` |
| Careflow調査証跡 | `.careflow/cases/<case_id>/evidence/`, `.careflow/cases/<case_id>/results/` |
| コードベース分析 | 大規模な影響範囲調査の成果物 |
| 設計事前調査 | Design doc の前段資料 |
| 実装前の影響範囲レポート | PR 前の技術調査ドキュメント |

**対象外**（短い事実ログで十分）:

- Plan ファイル（`.claude/plans/` 配下）
- 作業途中の中間メモ
- 特定の1ファイルについての軽い調査

非自明な調査では、最終レポートまたは要約を `.careflow` の result/evidence に保存する。

## 必須構造

詳細は `~/.claude/skills/investigation-report/SKILL.md` を参照。必須要素の抜粋:

1. **TL;DR** — 5-8項目の結論先出し
2. **対象読者の明示** — Qiita 風の前置き
3. **Part 構造** — 全体像 → 技術入門 → タスク → データフロー追跡 → 問題本質 → 前例探索 → 選択肢比較 → 推奨案 → 実装イメージ → 検証項目 → まとめ
4. **Mermaid 図最低 2枚** — アーキテクチャ / シーケンス / フロー
5. **比較表最低 1つ** — Markdown table
6. **ファイル:行番号** — 全引用に必須
7. **推測と事実の分離** — 明示的セクション分け
8. **付録** — 用語集 + 関連ファイル一覧

## 禁止事項

| 禁止 | 理由 |
|------|------|
| 「など」「他にも」で省略 | 調査の抜けを示唆、後続作業で再調査が発生 |
| ファイルパス・行番号なしの引用 | 事実検証ができない |
| 推測を事実として断言 | 実装時に詰む |
| コード全文貼り付け | 読み手のノイズ、メインコンテキスト圧迫 |
| 「確認していないが恐らく」等の言葉濁し | 「要検証」項目として明示化すべき |
| Part 見出しだけで中身薄い | 章立てが形骸化する |

`~/.claude/skills/investigation-report/templates/`

## 合理化対策

| 言い訳 | 反論 |
|--------|------|
| "時間がかかる" | 後続の実装・レビュー・引き継ぎ工数を削減する |
| "事実列挙で十分" | 事実を読んでも「次にどうすればいいか」が分からない調査は調査未了 |
| "読者を初心者に想定するのは冗長" | 半年後の自分・新規参加者に向けて書く。自分が忘れることを前提 |
| "選択肢比較は大袈裟" | 代替案との比較がない推奨は独断。根拠が示せない |

## 違反時の対応

レポートが基準を満たさない場合:

1. 作業を中断
2. 足りないセクションを追加
3. ファイルパス・行番号を補完
4. 推測と事実を分離
5. 再度提出
