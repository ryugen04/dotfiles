# ドキュメント作成・校正ガイドライン

## 概要

このガイドラインは、技術ドキュメントや記事作成において、AI生成らしさを排除し、自然で読みやすい文章を作成するための基準です。ドキュメント作成時の注意点と校正時のチェック項目を定義しています。

## 基本方針

### 1. 自然な日本語表現の重視
- 人間らしい自然な文体を維持する
- 機械的・定型的な表現を避ける
- 読み手にとって理解しやすい構成とする

### 2. 技術的正確性の確保
- 信頼できる情報源に基づく記述
- 最新情報の確認と反映
- 推測や不確実な情報の排除

## 禁止事項

### 1. 太字表現（*記法）の多用禁止

**禁止例：**
```
**重要なポイント** として、**データベース設計** では **正規化** が**必須**です。
```

**推奨例：**
```
データベース設計において正規化は必須の要素です。
```

**校正チェックポイント：**
- `**`記号が3回以上使用されていないか
- 強調が本当に必要な箇所のみに限定されているか

### 2. コロンの不適切な使用禁止

#### 見出し・タイトルでのコロン使用禁止

**禁止例：**
```
## データベース設計: 基本原則と実装方法
### パフォーマンス最適化: 効果的な手法
```

**推奨例：**
```
## データベース設計の基本原則と実装方法
### 効果的なパフォーマンス最適化手法
```

#### 文末でのコロン使用禁止

**禁止例：**
```
以下の点に注意が必要です:
主な特徴は次の通りです:
実装手順を説明します:
```

**推奨例：**
```
以下の点に注意が必要です。
主な特徴は次の通りです。
実装手順を説明します。
```

**校正チェックポイント：**
- 見出しに「：」「:」が含まれていないか
- 文末に「：」「:」が使用されていないか（箇条書き導入部含む）
- より自然な日本語表現に変更可能か

### 3. 過剰にフランクな口調・冗談の禁止

**禁止例：**
```
さてさて、今回はDockerについて解説しちゃいます！
めちゃくちゃ便利なツールなので、ぜひマスターしてくださいね♪
```

**推奨例：**
```
今回はDockerについて解説します。
コンテナ技術として広く活用されており、開発効率の向上に役立ちます。
```

**校正チェックポイント：**
- 「〜ちゃいます」「めちゃくちゃ」などの過度にカジュアルな表現
- 絵文字や記号の多用
- 技術内容にそぐわないユーモア表現

### 4. 同一表現の繰り返し禁止

**禁止例：**
```
自然言語処理は重要な技術です。自然言語処理を活用することで、自然言語処理の分野が発展します。
データベースの最適化により、データベースの性能が向上し、データベース運用が効率化されます。
```

**推奨例：**
```
自然言語処理は重要な技術です。この分野を活用することで、AI技術全体の発展に寄与します。
データベースの最適化により、システム性能が向上し、運用効率が大幅に改善されます。
```

**校正チェックポイント：**
- 同一の専門用語が3回以上連続して使用されていないか
- 同義語や代名詞で置き換え可能な表現があるか
- 文章全体の語彙に適度な多様性があるか

### 5. 機械的・定型的表現の禁止

**禁止例：**
```
以下に、重要なポイントを示します。
次に、具体的な手順を説明します。
最後に、まとめを記載します。
```

**推奨例：**
```
特に注目すべき点があります。
実装における具体的なアプローチを見ていきましょう。
これらの内容を踏まえると、以下のことが言えます。
```

**校正チェックポイント：**
- 「以下に〜」「次に〜」「最後に〜」の定型パターンが多用されていないか
- より自然で人間らしい文章表現に変更可能か
- 文章に個性や独自の視点が含まれているか

### 6. 推測ベースの情報記載禁止

**禁止例：**
```
この手法により、おそらく50%程度のパフォーマンス向上が期待できるでしょう。
開発チームの満足度も高まると思われます。
導入コストは比較的安価になる傾向があります。
```

**推奨例：**
```
公式ベンチマークによると、この手法で平均40-60%のパフォーマンス向上が実測されています。
（参考: https://example.com/benchmark-results）
```

### 7. 不自然な定義・説明の挿入禁止

**禁止例：**
```
Reactは人気のライブラリです。Reactとは、ユーザーインターフェースを構築するためのJavaScriptライブラリのことです。このReactを使用することで、効率的な開発が可能になります。
```

**推奨例：**
```
Reactは、ユーザーインターフェース構築に特化したJavaScriptライブラリとして、多くの開発者に支持されています。
```

**校正チェックポイント：**
- 文章の流れを無視した突然の定義挿入がないか
- 説明が文脈に自然に組み込まれているか
- 読者が既に知っている可能性のある基本概念の過度な説明がないか

### 8. 感情・経験の欠如した表現の回避

**禁止例：**
```
この技術は有用です。効率的に動作します。実装は簡単です。
結果として、システムが改善されます。
```

**推奨例：**
```
この技術を実際に導入してみると、その威力を実感できます。
想像以上にスムーズに動作し、開発チームからも好評を得ています。
```

**校正チェックポイント：**
- 単調で事務的な表現が連続していないか
- 実体験や具体的な感想が適度に含まれているか
- 読み手の共感を呼ぶ表現があるか

## 文体・表現に関する規則

### 1. 体言止めの制限

**禁止例：**
```
システムの高度化。
セキュリティの強化。
ユーザビリティの向上。
```

**推奨例：**
```
システムを高度化します。
セキュリティを強化する必要があります。
ユーザビリティの向上を図ります。
```

**校正チェックポイント：**
- 連続する体言止めがないか
- より自然な述語表現に変更可能か

### 2. 箇条書きの適切な使用

**過剰な例：**
```
## Dockerの特徴
- 軽量性
- 可搬性  
- スケーラビリティ
- 開発効率性
- デプロイメント簡素化
```

**推奨例：**
```
## Dockerの特徴
Dockerには軽量性と可搬性という大きな特徴があります。コンテナは仮想マシンと比較してオーバーヘッドが少なく、異なる環境間でも同一の動作を保証します。また、スケーラビリティに優れ、必要に応じてコンテナの増減が容易に行えます。
```

**校正チェックポイント：**
- 箇条書きが5項目以上連続していないか
- 文章による説明がより適切でないか

### 3. 一人称の統一

**技術ドキュメント：** 「筆者」または一人称なし
**個人ブログ：** 「私」で統一
**チュートリアル：** 一人称なしまたは「我々」

**校正チェックポイント：**
- 文書全体で一人称表現が統一されているか
- ドキュメントの性質に適した表現になっているか

## 技術情報の扱い

### 1. 情報源の明記必須

**必須要素：**
- 公式ドキュメントへのリンク
- 信頼できる技術サイトの参照
- バージョン情報の明記
- 更新日時の確認

**記載例：**
```
React 18.2.0では新しいConcurrent Featuresが導入されました。
詳細は公式ドキュメントを参照してください。
（参考: https://react.dev/blog/2022/03/29/react-v18 - 2022年3月更新）
```

### 2. 最新性の確保

**校正チェックポイント：**
- 参照している情報が1年以内のものか
- 廃止予定（deprecated）の技術を推奨していないか
- バージョン情報が明記されているか

## 具体例の重視

### 1. 抽象的説明の回避

**禁止例：**
```
効率的なアルゴリズムを実装することで、システム全体のパフォーマンスが向上します。
```

**推奨例：**
```
バブルソートをクイックソートに変更することで、10,000件のデータソートが3秒から0.1秒に短縮されました。

// 実装例
function quickSort(arr) {
  if (arr.length <= 1) return arr;
  // ... 具体的なコード
}
```

### 2. 実行可能なコード例の提供

**校正チェックポイント：**
- コード例が実際に動作するか
- 必要な依存関係が明記されているか
- エラーハンドリングが適切に含まれているか

## 校正時の総合チェックリスト

### 文体・表現
- [ ] 太字（**）が3回以上使用されていない
- [ ] 見出しや文末にコロンが使用されていない
- [ ] 過度にカジュアルな表現がない
- [ ] 体言止めが連続していない
- [ ] 一人称表現が統一されている
- [ ] 同一の専門用語が不自然に繰り返されていない
- [ ] 機械的・定型的な表現が多用されていない
- [ ] 不自然な定義や説明の挿入がない
- [ ] 感情や経験を感じられる自然な表現になっている

### 技術的正確性
- [ ] すべての技術情報に信頼できる情報源が付いている
- [ ] 参照情報が最新（1年以内）である
- [ ] 推測ベースの記述がない（「おそらく」「思われます」「傾向があります」など）
- [ ] 具体的な数値や事例に裏付けがある
- [ ] バージョン情報が明記されている

### 構成・可読性
- [ ] 箇条書きが適切に使用されている
- [ ] 具体例やコード例が含まれている
- [ ] 段落構成が論理的である
- [ ] 読み手の立場で理解しやすい順序になっている

### AI生成らしさの排除
- [ ] 機械的・定型的な表現がない
- [ ] 同一表現の不自然な繰り返しがない
- [ ] 突然の定義挿入など文脈を無視した説明がない
- [ ] 人間らしい自然な文章になっている
- [ ] 独自の視点や経験が含まれている（適切な場合）
- [ ] 感情や実体験を感じられる表現がある

## 実施方法

1. **作成時：** このガイドラインに従って文書を作成
   - 特に同一表現の繰り返しや定型的な文章パターンに注意
   - 人間らしい自然な表現を心がける
2. **校正時：** チェックリストを用いて全項目を確認
   - AI生成らしさの特徴的パターンを重点的にチェック
3. **修正時：** 指摘された項目について具体的な改善案を提示
   - 機械的な表現を人間らしい表現に置き換える

このガイドラインに従うことで、自然で読みやすく、技術的に正確なドキュメントの作成と校正が可能になります。特に2024-2025年の日本のエンジニア界隈で広く指摘されているAI生成記事の特徴（同一表現の繰り返し、機械的な文章パターン、感情の欠如など）を回避し、人間らしい自然な文章を作成することが重要です。
