# レトロスペクティブ準備

スプリントレトロスペクティブの準備を行う。
データ収集、手法選定、アジェンダ作成を支援。

## レトロスペクティブの目的

スクラムガイドより:
> スプリントレトロスペクティブの目的は、品質と効果を高める方法を計画することである。

チームが以下を検査する:
- 個人に関して
- 相互作用に関して
- プロセスに関して
- ツールに関して
- 完成の定義に関して

## 入力

1. **スプリント番号**: Sprint N
2. **スプリント期間**: 開始日〜終了日
3. **前回の改善アクション**: 前回レトロで決めたアクション一覧
4. **スプリント中のメモ**: 気づいたこと、問題、良かったこと（任意）
5. **メトリクス**: 完了PBI数、未完了PBI数、ベロシティ（任意）

## レトロ手法の選定基準

| 状況 | 推奨手法 |
|------|---------|
| 初回・基本 | KPT |
| 行動変容にフォーカス | Start/Stop/Continue |
| 学びを重視 | 4Ls (Liked/Learned/Lacked/Longed for) |
| 感情面を扱いたい | Mad/Sad/Glad |
| 方向性・ビジョン議論 | Sailboat |
| 長期の振り返り | Timeline |
| 行動の優先度議論 | Starfish |
| マンネリ打破 | 前回と違う手法を選ぶ |

## 手法詳細

### KPT
- **Keep**: 続けたいこと
- **Problem**: 問題・課題
- **Try**: 試したいこと

### Start/Stop/Continue
- **Start**: 始めること
- **Stop**: やめること
- **Continue**: 続けること

### 4Ls
- **Liked**: 良かったこと
- **Learned**: 学んだこと
- **Lacked**: 足りなかったこと
- **Longed for**: 欲しかったこと

### Mad/Sad/Glad
- **Mad**: 怒り・苛立ち
- **Sad**: 悲しみ・残念
- **Glad**: 喜び・満足

### Sailboat
- **Wind (追い風)**: チームを前進させる力
- **Anchor (錨)**: チームを停滞させるもの
- **Rocks (岩礁)**: 潜在的なリスク
- **Island (島)**: 目指すゴール

### Timeline
1. スプリントの出来事を時系列で並べる
2. 感情の起伏をマッピング
3. 転換点を特定
4. 学びを抽出

## 出力フォーマット

```markdown
# Sprint {N} レトロスペクティブ準備

## スプリント概要
- **期間**: {開始日} - {終了日}
- **完了PBI**: {数}件
- **未完了PBI**: {数}件（理由: ...）
- **ベロシティ**: {SP} (前回比: +/-{差分})

## 前回アクションの振り返り
| アクション | 担当 | 状態 | コメント |
|-----------|------|------|---------|
| {アクション1} | {誰} | 完了/継続中/未着手 | {補足} |
| {アクション2} | {誰} | ... | ... |

## 今回の手法提案

### 推奨: {手法名}

**選定理由**: {チームの状況や今スプリントの特徴に基づく理由}

**進め方**:
1. チェックイン（5分）: 今の気持ちを一言で
2. データ収集（10分）: {手法に応じた活動}
3. インサイト生成（10分）: パターンや根本原因を議論
4. アクション決定（10分）: 次スプリントで試すこと
5. クロージング（5分）: レトロのレトロ

**準備物**:
- {必要なツールや準備}

## 議論候補トピック

スプリント中の観察から:
- {トピック1}: {背景}
- {トピック2}: {背景}

## ファシリテーターへのメモ
- {注意点やTips}
```

## 保存先

```
.claude/scrum/retro/{YYYYMMDD}-sprint-{N}.md
```

**注意:** このファイルはプロジェクト固有です。
プロジェクトルートの相対パスで保存されます。

例: `.claude/scrum/retro/20260118-sprint-2.md`

## 注意事項

- レトロの準備は「事前の議題設定」ではない
- チームが自由に発言できる場を作ることが重要
- 準備資料はファシリテーターの参考用
- 改善アクションは1〜3個に絞る（多すぎると何も進まない）
- 全員が発言できる工夫を入れる
