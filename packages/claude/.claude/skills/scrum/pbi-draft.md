# PBI案作成

POからの曖昧な要望を、リファインメントに持ち込むためのPBI案に変換する。

## 入力

ユーザーから以下のいずれかを受け取る:
- POからの要望（テキスト、Slackコピペ等）
- 口頭での機能説明メモ
- 既存機能の改善要望

## 処理手順

### Step 1: 要望の理解
- 誰のための機能か（役割）
- 何を実現したいか（機能）
- なぜ必要か（価値・目的）

### Step 2: ユーザーストーリー形式への変換
```
As a [役割],
I want [機能],
So that [価値].
```

### Step 3: 受け入れ基準の定義
Gherkin形式で記述:
- 正常系シナリオ（最低1つ）
- 異常系・エッジケースシナリオ（最低1つ）
- 境界値シナリオ（該当する場合）

### Step 4: DoRチェック
Definition of Readyの各項目について、満たしているか/未確認かを判定。

### Step 5: 不明点・調査必要事項の洗い出し
Spikeが必要な場合は明示。

## 出力フォーマット

```markdown
# PBI案: {日本語タイトル}

## ユーザーストーリー
As a [役割],
I want [機能],
So that [価値].

## 背景・コンテキスト
{この機能が必要になった背景}

## 受け入れ基準
- [ ] **Scenario 1: {正常系タイトル}**
  - Given: {前提条件}
  - When: {操作・イベント}
  - Then: {期待結果}

- [ ] **Scenario 2: {異常系タイトル}**
  - Given: {前提条件}
  - When: {操作・イベント}
  - Then: {期待結果}

- [ ] **Scenario 3: {境界値/エッジケース}**
  - ...

## スコープ外（やらないこと）
- {明示的にスコープから除外するもの}

## DoRチェック
### 確認済み
- [x] ユーザーストーリーが明確
- [x] ACが定義されている

### 要確認（リファインメントで議論）
- [ ] デザイン確認
- [ ] 技術的実現可能性
- [ ] 依存関係
- [ ] 見積もり

## 調査が必要な事項
- {不明点、Spikeが必要な場合はその内容}

## メモ
- {その他気づいた点}
```

## 保存先

```
.claude/scrum/backlog/{YYYYMMDD}-{日本語タイトル}.md
```

**注意:** このファイルはプロジェクト固有です。
プロジェクトルートの相対パスで保存されます。

例: `.claude/scrum/backlog/20260118-ユーザー認証機能.md`

## 注意事項

- これは「案」であり、リファインメントでチームが修正することを前提とする
- 完璧を目指さない。議論のたたき台として十分な品質でよい
- 技術的な実装詳細には踏み込まない（それはプランニング後）
- 見積もりは行わない（チームで行う）
