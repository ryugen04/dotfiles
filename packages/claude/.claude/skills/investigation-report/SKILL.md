# investigation-report

調査レポートを「初見の人が読んで理解できる Qiita記事」のレベルまで丁寧に書くためのスキル。事実の列挙ではなく、**読み手の理解を段階的に組み立てる構成**を強制する。

## トリガー例

以下のような作業で発動する:

- 調査レポートを書く（`.md` ファイル、`.claude/work/` 配下が典型）
- コードベース分析ドキュメントを書く
- 技術設計の事前調査・design doc の前段資料を書く
- 実装前の影響範囲調査レポートを書く
- 「調査結果をまとめて」「レポートにして」「調査ドキュメントを作って」とユーザーが依頼した時

## 必須構造（テンプレート）

全ての調査レポートは以下の構造を踏襲する。セクション順序は入れ替え可だが、**項目の省略は禁止**。

### 冒頭

```markdown
# {タイトル} — {1行の副題（何が分かる記事か）}

> **対象読者**: （例: プログラミング初心者、このリポジトリに不慣れな人、など）
>
> **この記事で学べること**: 読者が得られる具体的な理解を1-3行

---

## TL;DR（先に結論だけ）

- 箇条書きで5-8項目
- 調査結果の核心
- 推奨案があれば明示
- なぜそうなるかは本文で、と誘導
```

### 本文構成（Part 方式）

```markdown
## Part 1: まずは地図を広げよう — 全体像

### 1.1 対象システムは何か
システムの用途・ユーザー像

### 1.2 構成要素
マルチリポジトリ・レイヤー・主要コンポーネント
（Mermaid 図必須）

### 1.3 重要な前提事実
読者がこの記事を理解する上で欠かせない背景
```

```markdown
## Part 2: 技術スタック超入門

細かく知らなくて大丈夫。「何となくこういうもの」を掴めば OK、と断る。

### 2.1 {言語/フレームワーク1}
超シンプルな例1つ + プロジェクト内での位置づけ

### 2.2 {言語/フレームワーク2}
...
```

```markdown
## Part 3: 今回のタスクは何か

### やりたいこと
要求をユーザー視点で1-3文

### なぜ特別か / なぜ自明でないか
背景・制約・歴史的経緯
```

```markdown
## Part 4-N: 現状の流れを追う

### 画面の入り口
ファイルパスから出発

### UI コンポーネント
何をどう表示しているか

### 状態管理 / Store
データの持ち方

### リクエスト発行
APIリクエストの具体例 + 実コード引用

### サーバ側の受け付け
Resolver / Controller

### 業務ロジック
UseCase / Service

### 永続化
DB / ファイル / 外部API
```

```markdown
## Part: 問題の本質

現状の何が「このタスクを難しくしているか」を1-2段落で。
根本原因をピンポイントで指摘する。
```

```markdown
## Part: 既存の前例・類似実装を探す

**この Part は必須**。「似た場所で既にやっていないか」を探して書く。
前例があれば、そのファイルパス・実装パターンを引用する。
前例が無ければ「探したが無かった」と明記する。
```

```markdown
## Part: 解決策の選択肢

### 選択肢 A/B/C/D... を列挙
各選択肢について:
- 概要（1-2文）
- スコープ（触るファイル/リポジトリ）
- コスト（小/中/大/超大）
- リスク
- 推奨度

比較表（Markdown table）で可視化する。
```

```markdown
## Part: 推奨案と決定要因

- どれを推奨するか
- なぜそれか（決定要因を3つ程度）
- 却下した案の却下理由（合理化ではなく根拠）
```

```markdown
## Part: 実装イメージ

推奨案で触るファイル一覧と、変更内容の概念コード（Before/After）。
実コードの行番号を明示する。
```

```markdown
## Part: 不確実性・検証項目

**この Part は必須**。推測で書いた部分、追加調査が必要な部分を明示。
箇条書きで5項目程度。
```

```markdown
## Part: まとめ

### 今回学んだこと
読者が得られる一般化可能な知見

### 今回の結論
タスク固有の結論を短く

### 説明できないなら、まだ終わっていない
読者が自分の言葉で説明できるべきポイントを2-3個列挙
```

### 付録（必須）

```markdown
## 付録: 用語集

| 用語 | 意味 |
|------|------|
| ... | ... |

## 付録: 関連ファイル一覧

### フロント
- `path/to/file.ts:L行番号` — 役割

### バックエンド
- `path/to/file.kt:L行番号` — 役割
```

## スタイル規則

### 必須

- **Mermaid 図 を最低 2つ**: アーキテクチャ図・シーケンス図・フローチャートのどれか
- **比較表（Markdown table）を最低 1つ**
- **ファイル:行番号** を全ての引用に付ける（`path/to/file.ts:L42-67` 形式）
- **コード引用は最大 15行** 、それ以上は概念コード（疑似コード）で表現
- **推測と事実の分離**: 事実パートと分析・仮説パートを明示的に分ける
- **初見の読者想定の語り**: 「〇〇って何?」のような問いかけを冒頭で挟む
- **比喩の活用**: 技術的な仕組みを日常的な比喩で補強する（契約書・橋渡し・地図 等）
- **段階的な理解誘導**: 前の Part で説明したことを次の Part で自然に使う
- **結論の言い切り**: 「〜と思われる」「〜かもしれない」を避け、事実は言い切る。不確実なら「検証が必要」と明記

### 推奨

- **軽い強調表現** を要所で使う（例: 「**ここが問題だッ！**」「**油断するなッ！**」「**なんという〇〇ッ！**」）
  - ただし**連発しない**。1 Part に 1-2 回まで
  - 派手さで誤魔化さず、本文の論理構造は硬派に保つ
- **疑問→回答の対話形式**: 「なぜ?」と問いかけて次の段落で答える
- **図示 > 箇条書き > 文章** の優先順位で情報を届ける

### 禁止

- **「など」「他にも」「以下同様」で省略**: 重要な情報は全部書く
- **ファイルパス・行番号の省略**: 必ず付ける
- **「確認していないが恐らく」などの言葉濁し**: 「要検証」として項目化する
- **推測を事実として書く**: 明確に分離
- **コード全文の貼り付け**: 引用は最大15行
- **AI 生成臭い定型表現**: 「〜することが重要です」「〜を理解することで〜」等の抽象表現を連発しない
- **見出しだけの Part**: 中身のない章立てを作らない

## 粒度の目安

| 項目 | 目安 |
|------|------|
| 全体の長さ | 6000-15000字（調査の深さに応じて） |
| Part 数 | 8-12 |
| Mermaid 図 | 2-5 枚 |
| 比較表 | 2-4 表 |
| コード引用 | 5-15 箇所 |

## 出力先

- 保存先: プロジェクトの `.claude/work/{topic}/` 配下（プロジェクトルールに従う）
- ファイル名: `article-{theme}.md` または `investigation-{theme}.md`
- 既存の短い調査ログ（`investigation-codebase.md` 等）がある場合は、それを踏襲して **別ファイル** として Qiita 記事版を作る（上書きしない）

## 合理化対策

| 言い訳 | 反論 |
|--------|------|
| "事実列挙で十分" | 読み手が「地図を描ける」ようになるのが調査レポートの目的 |
| "この長さは冗長" | 粒度の目安を守れば冗長ではない。読み手が「全体像→詳細→選択肢→推奨」を1本で辿れる |
| "初心者向けは不要" | 半年後の自分が初心者。未来の自分のために書く |
| "時間がかかる" | 後続の実装・レビュー工数を削減するので総工数は減る |

## 違反時の対応

レポートがこの構造を満たさない場合:

1. 作業を中断
2. 足りない Part を追加
3. 省略された引用に行番号を補う
4. 推測と事実を分離
5. 再度ユーザーに提出
