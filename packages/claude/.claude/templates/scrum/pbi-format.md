# PBI (Product Backlog Item) フォーマット

## ファイル名規約

```
.claude/scrum/backlog/{YYYYMMDD}-{日本語タイトル}.md
```

例: `20260129-ユーザーパスワードリセット機能.md`

## テンプレート

```markdown
# {PBIタイトル}

## ステータス

Draft / Ready / In Progress / Done

## ユーザーストーリー

**{ペルソナ}** として、
**{行動・機能}** したい。
なぜなら **{価値・理由}** だから。

## 受け入れ基準（Acceptance Criteria）

### AC1: {基準名}
- Given: {前提条件}
- When: {アクション}
- Then: {期待される結果}

### AC2: {基準名}
- Given: {前提条件}
- When: {アクション}
- Then: {期待される結果}

## DoR（Definition of Ready）チェックリスト

- [ ] ユーザーストーリーが明確
- [ ] 受け入れ基準が定義済み
- [ ] 依存関係が特定されている
- [ ] 1スプリント内で完了可能なサイズ
- [ ] チームが実装可能と判断

## 依存関係

- {前提PBI、外部依存等}

## 技術的メモ

{実装時に考慮すべき技術的な点}

## 見積もり

- ストーリーポイント: {未見積もり / 1 / 2 / 3 / 5 / 8 / 13}

## 未解決事項

- [ ] {リファインメントで確認が必要な点}

## 更新履歴

| 日付 | 内容 |
|------|------|
| {YYYY-MM-DD} | 初版作成 |
```

## 書き方のコツ

### ユーザーストーリーのポイント
- ペルソナは具体的に（「ユーザー」より「管理者」「新規会員」など）
- 「〜機能がほしい」ではなく「〜したい」で表現
- 価値・理由が最も重要

### 受け入れ基準のポイント
- テスト可能な形で記述
- 曖昧な表現を避ける（「適切に」「正しく」は禁止）
- ハッピーパスだけでなくエラーケースも

### サイズの目安
- 1スプリント内で完了できるサイズ
- 大きすぎる場合は分割を検討
- 見積もりが13以上なら分割必須
