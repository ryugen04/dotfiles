# Definition of Ready (DoR)

PBIがプランニングに持ち込まれるための条件。
全ての項目にチェックが入っていること。

---

## 1. 価値の明確さ

- [ ] ユーザーストーリー形式で記述されている
      ```
      As a [役割],
      I want [機能],
      So that [価値].
      ```
- [ ] POがビジネス価値を説明できる
- [ ] なぜ今このスプリントでやるべきか説明できる

## 2. 範囲の明確さ

- [ ] 受け入れ基準（AC）が3つ以上定義されている
- [ ] 正常系シナリオが含まれている
- [ ] 異常系・エッジケースシナリオが含まれている
- [ ] スコープ外（やらないこと）が明示されている
- [ ] QAがACをレビュー済み

## 3. デザイン

- [ ] UIモックまたはワイヤーフレームがある（UI変更を伴う場合）
- [ ] デザイナーがレビュー済み
- [ ] インタラクションが定義されている（必要な場合）

## 4. ドメイン理解

- [ ] 必要な業務知識が整理されている
- [ ] 用語の定義が明確
- [ ] 不明点がある場合はSpikeとして分離済み

## 5. 実現可能性

- [ ] 技術的に実現可能と開発者が判断している
- [ ] 外部依存（API、他チーム、外部サービス等）が特定されている
- [ ] 依存関係が解決済み、または解決の見通しがある
- [ ] 1スプリント内で完了可能な粒度である

## 6. 見積もり

- [ ] チームで見積もり済み
- [ ] 見積もり手法: ストーリーポイント / 時間 / Tシャツサイズ（チームで選択）

---

## DoRの運用

### リファインメントで確認
- 各PBIについてDoRチェックを行う
- 全項目が満たされればReady
- 満たされない項目があれば、次のリファインメントまでに解決

### 例外処理
- 緊急のバグ修正等、DoRを完全に満たせない場合もある
- その場合はチームで合意の上、リスクを認識して進める
- レトロで振り返り、改善につなげる

### 継続的改善
- DoRは固定ではない
- レトロで見直し、チームに合った形に調整する
- 項目の追加・削除はチームで合意して行う
